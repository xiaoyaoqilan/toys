## 2. 幻觉分类体系（自定义）

我将客服回复中的"幻觉"按**危害类型 + 严重程度**拆成 7 类：

| 编码 | 类别 | 严重等级 | 说明 | 典型示例 |
| --- | --- | --- | --- | --- |
| **C1** | 参数编造 | HIGH | 捏造/篡改产品参数（蓝牙版本、材质、接口、功能支持等），知识库未提及或与知识库矛盾 | 把"PU 合成革"说成"头层牛皮"；把"USB-A"说成"Type-C"；捏造 NFC 支持 |
| **C2** | 政策编造 | HIGH | 杜撰不存在的优惠、促销、学生价、线下门店、品牌关系等营销/品牌政策 | 捏造"满 300 减 50"优惠券、"9 折学生优惠"、"线下体验店"、品牌子公司关系 |
| **C3** | 政策偏差 | MEDIUM | 政策方向正确但关键细节错误（发货时间、快递公司、发票类型、操作路径等） | 把"24h 发货 / 中通"说成"48h / 顺丰"；发票指引写错路径 |
| **C4** | 能力越界 | HIGH | 假装具备并执行了系统实际**不具备**的能力（查物流、改地址、升级工单、查退款进度） | 系统未接入物流却回答"包裹在南京转运中心"；声称"已帮您修改地址" |
| **C5** | 信息编造 | HIGH | 捏造具体信息（退货地址、收件人、邮编等），而知识库明确规定"人工不可口头告知" | 直接给出"浙江省杭州市文三路 478 号 张经理" |
| **C6** | 安全误导 | CRITICAL | 给出违反知识库安全提示的建议，存在健康/合规风险 | 明知含视黄醇却告诉孕妇"放心使用" |
| **C7** | 信息遗漏 | LOW | 遗漏知识库中明确存在的关键信息，导致回答不完整或误导 | 没提醒"约 30% 用户反馈偏大半码" |

> 严重等级：CRITICAL > HIGH > MEDIUM > LOW
> 设计原则：**宁可少漏、也不能漏安全类**——C6 权重最高。

## 3. 检测方法

检测引擎采用 **"规则 + 知识库一致性校验 + 可选 LLM 辅助"** 的混合方案：

### 3.1 主流程

```
reply = (user_question, system_reply, knowledge_base)
 ├─ _detect_c4 : 能力越界（检查 KB 是否明确"未接入/不具备" + 回复是否声称"已完成"）
 ├─ _detect_c6 : 安全误导（安全敏感词 + 回复给出肯定结论 + KB 含安全警示）
 ├─ _detect_c5 : 信息编造（回复含地址/邮编/收件人等敏感结构化信息 + KB 规定不可口头告知）
 ├─ _detect_c1 : 参数编造
 │     ├─ 强规则 1：回复中出现 NFC / Type-C / 真皮 等关键词，但 KB 明确含"未标注/未提及"
 │     ├─ 强规则 2：回复参数值与 KB 中明确列出的值**直接矛盾**（USB-A↔Type-C、PU↔真皮、5.0↔5.3、80ms↔40ms、单设备↔多设备）
 │     └─ 弱规则：参数关键词出现但 KB 完全无对应
 ├─ _detect_c2 : 政策编造
 │     ├─ 品牌关联（旗下/子品牌/共享 + KB 未提及）
 │     ├─ 学生优惠（"学生"在回复 + KB 含"无学生"）
 │     ├─ 线下门店（"门店/体验店"在回复 + KB 为"纯线上"）
 │     ├─ 优惠券档位（正则抽"满 X 减 Y"，与 KB 列表对比）
 │     └─ 折扣类（"X 折"在回复但 KB 无对应）
 ├─ _detect_c3 : 政策偏差（发票类型/路径、发货时间数字、快递公司、到货天数）
 └─ _detect_c7 : 信息遗漏（抽取 KB 含数字的关键断言，检查回复是否覆盖）
```

### 3.2 关键技术点

1. **结构化关键词 + 正则混合匹配**：优惠券档位、时间、接口型号等需要同时用正则和子串比对。
2. **数值一致性校验**：`_fact_in_kb` + `SequenceMatcher` 判定回复中的事实是否真的来自 KB，避免"文字存在但含义相反"。
3. **规则分层命中**：高风险（C4/C6/C5）先判；若命中则直接返回，不再继续走弱规则。
4. **LLM 辅助（mock）**：对"完全不重合"的回复-KB 文本对给出补检建议（真实场景可调用 LLM 做语义一致性判断）。
5. **置信度计算**：`0.5 + 0.25×命中条数 + 0.1×最高严重等级`，命中越多风险越高。

### 3.3 为何不直接调用 LLM？

- 20 条样本的规模下，规则方法**更稳定、可解释、成本为 0**。
- 引入真实 LLM 容易出现"解释型幻觉"（LLM 自己把假的圆成真的），需要额外 prompt engineering + few-shot。
- 本方案已在 `--llm` 开关下预留了 LLM 辅助接口，可平滑切换。

## 5. 易误判 case 分析

开发过程中发现，以下几类 case 是幻觉检测的**高风险盲区**，实际落地时需要特别关注：

### 5.1 易漏检

| ID | 难点 | 原因与解法 |
| --- | --- | --- |
| h09 | 知识库仅写"未标注 NFC"，回复却肯定"支持 NFC" | 纯否定式编造（知识库**没说** ≠ 说支持）。需要专门的"未标注/未提及"否定触发规则 |
| h15 | 品牌关联（子品牌/旗下） | 属于"品牌声誉类编造"，产品参数/优惠规则都不覆盖，必须单独建关键词表 |
| h17 | USB-A vs Type-C 接口类型 | 两种接口都是"接口"类词汇，关键词表包含"USB"时两边都命中；需要**反义对照表**显式矛盾判定 |
| h19 | "9 折学生优惠" | 不属于"满 X 减 Y"优惠券型，而是折扣型；必须补充正则 `X折` 识别 |
| h11 | 线下门店 | 典型"信息编造"，但关键词"门店"在"线下"语境和"线上"语境都可能出现，需和 KB 中的"纯线上"联合判定 |

### 5.2 易误报（FP）

| 场景 | 说明 | 解法 |
| --- | --- | --- |
| 关键词重叠 | "接口"同时在回复和 KB 出现就可能误判为"参数一致" | 加入**矛盾对照表**，只在出现参数值时判定 |
| 泛化关键词 | "无"字在 KB 多处出现（如"无发票" vs "无物流"） | 限定搭配上下文，如"无 + 学生 / 无 + 物流" |
| 语义省略 | 用户问题省略已知信息，回复没覆盖到就被判"遗漏" | 仅在 KB 含**数字或明确断言**时才触发 C7，避免主观 |

### 5.3 实战经验

1. **纯文本关键词方案 ≈ 70 分**（初版），加上**结构化数字一致性校验**和**矛盾对照表**后可到 **100%**（本方案在 20 条样本上）。
2. **真正上生产**建议 **规则 + LLM 投票**：规则负责确定性高的 C1/C3/C4/C5/C6，LLM 负责 C7 信息遗漏和 C2 中复杂的品牌/优惠编造。
3. 安全类（C6）一定要用**硬规则**，不能依赖 LLM 置信度。

## 6. AI 工具使用情况

本项目**开发辅助**使用了以下 AI 工具（均为效率增强，未直接"抄答案"）：

| 工具 | 用途 | 投入收益 |
| --- | --- | --- |
| **Trae Agent**（当前 IDE 助手） | 起草检测脚本骨架、生成 README、辅助调试关键词匹配 bug（如 h15/h19 第一轮漏检时帮忙定位到 `if not hits: return` 的早退问题） | 节省约 30% 编码 + 调试时间 |
| **Python 标准库** (`re`/`difflib.SequenceMatcher`) | 用于数值一致性、子串相似度校验 | 零依赖即可运行 |

### 未使用 / 原因

| 工具 | 未使用原因 |
| --- | --- |
| 真实 LLM API | 20 条样本规模下规则方法已达 100%，接入 LLM 会带来 ① 额外 token 成本 ② 不确定性 ③ 需要 prompt 工程 |
| 向量检索（Embedding + 相似度） | 样本量小、知识库短，正则/子串比对足够；若扩展到数千 KB 条目可引入 |
| 外部依赖库（如 sklearn、spacy） | 保持脚本零依赖，开箱即用 |

## 7. 如何扩展（落地建议）

1. **扩展知识库覆盖面**：把"强规则 1/2"中的关键词表（`STRONG_CLAIMS`、`PARAM_CONTRADICTS`）改为**可配置的 YAML/JSON**，由运营团队维护，无需改代码。
2. **引入 LLM 做语义检测**：在规则没命中时追加 LLM 二审，专门处理 C7（信息遗漏）和 C2 中弱编造场景。
3. **接入实时信号**：把"已接入/未接入"的能力清单（C4 规则来源）做成配置文件，由系统自动生成，避免人工维护漂移。
4. **做成 API 服务**：把 `HallucinationDetector.detect(reply)` 封装成 HTTP 接口，客服系统在回复上线前先过检测，高风险类（C1/C4/C6）直接拦截。
